# 0007 — Courses (minimal model + permissions)

- **Status:** accepted (initial CRUD shipped — uncommitted)
- **Date:** 2026-05-08

## Context

Phase 1's reason for existing is the university workflow:

```
INSTRUCTOR  creates a COURSE
INSTRUCTOR  attaches problems to it (the syllabus)
INSTRUCTOR  enrols students into a GROUP for the course
INSTRUCTOR  schedules an EXAM that picks problems from the syllabus
STUDENT     sees the course, solves the problems, takes the exam
ADMIN       can do all of the above on any course on the installation
```

This ADR settles the **course** primitive. Groups, exams and gradebook
are separate ADRs / commits later in Phase 1 — they all reference the
courses table from here.

A bare-minimum course is enough for the AITU pilot:

- A short, human-typed `slug` to put in URLs and exam handouts.
- A title and a markdown-ish `description`.
- An owner (the instructor who created it).
- A list of problems attached to it, in instructor-controlled order.

We do *not* need release dates, prerequisites, multi-instructor
co-teaching, visibility flags, or course archives in the first cut.
Each of those is a follow-up if a customer actually asks for it.

## Decision

### Tables

```sql
CREATE TABLE courses (
  id          INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  slug        TEXT NOT NULL UNIQUE,
  title       TEXT NOT NULL,
  description TEXT,
  owner_id    INTEGER NOT NULL REFERENCES users(id),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE course_problems (
  course_id  INTEGER NOT NULL REFERENCES courses(id)  ON DELETE CASCADE,
  problem_id INTEGER NOT NULL REFERENCES problems(id) ON DELETE CASCADE,
  position   INTEGER NOT NULL DEFAULT 0,
  added_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (course_id, problem_id)
);
```

`course_problems.position` is a simple integer ordering field (0, 1,
2, ...). We do not enforce uniqueness on `(course_id, position)` — two
problems can sit at "0" and the UI will then sort by `(position,
added_at, problem_id)` for stable ordering. This is on purpose: making
position unique would force every reorder to do an N-row update, and
the UI for reordering can just hand back ascending positions when the
instructor clicks Save.

### Foreign-key behaviour

- `courses.owner_id → users(id)` — **NO action** on user delete.
  Postgres' default for `REFERENCES` is `NO ACTION`, which means you
  cannot delete a user that owns courses. A future admin UI for
  "delete user" must offer to reassign their courses to another
  INSTRUCTOR/ADMIN before actually deleting the user. This is the
  conservative choice; we'd rather block a delete than orphan a
  course.
- `course_problems.{course,problem}_id → ... ON DELETE CASCADE` —
  cleaning up the link table when either side disappears is always
  what you want. No syllabus row should ever point at a problem that
  no longer exists.

### Permissions

| Action                                     | Required role/relation       |
|--------------------------------------------|------------------------------|
| `GET /api/courses`                         | any authenticated user       |
| `GET /api/courses/:slug`                   | any authenticated user       |
| `POST /api/courses`                        | INSTRUCTOR or ADMIN          |
| `PUT /api/courses/:slug`                   | owner or ADMIN               |
| `DELETE /api/courses/:slug`                | owner or ADMIN               |
| `POST /api/courses/:slug/problems`         | owner or ADMIN               |
| `DELETE /api/courses/:slug/problems/:pSlug`| owner or ADMIN               |

The route layer enforces *the role half* via `requireRole(INSTRUCTOR,
ADMIN)`. The owner check happens **inside the service** so every
caller (HTTP, future CLI, future admin script) goes through it. This
mirrors the pattern from ADR 0006 §setRole.

ADMIN is universally allowed because they are the installation owner
(see ADR 0006). An INSTRUCTOR who is not the course owner cannot
modify it, even though they share the role with the owner. This
matches the AITU expectation that one professor "owns" their course.

### Visibility (read-side)

In this commit, **every authenticated user can list and read every
course**. We deliberately do NOT scope by enrolment yet — the
enrolment table doesn't exist in this commit. That visibility model
is revisited when groups land:

- STUDENT → sees only courses they're enrolled in via a group.
- INSTRUCTOR → sees their own courses + courses they assist on.
- ADMIN → sees all.

The current "all authenticated → all courses" is acceptable for a
Phase 1 development setup but **must** be tightened before the AITU
pilot ships. Groups module will replace the route handler, not just
add a filter, so the wider read model doesn't accidentally leak in
production.

### URL shape

```
GET    /api/courses                       — list (read-only)
GET    /api/courses/:slug                 — detail incl. problems list
POST   /api/courses                       — create (INSTRUCTOR/ADMIN)
PUT    /api/courses/:slug                 — update (owner-or-ADMIN)
DELETE /api/courses/:slug                 — delete (owner-or-ADMIN)
POST   /api/courses/:slug/problems        — attach problem
DELETE /api/courses/:slug/problems/:pSlug — detach problem
```

Slugs only; we never expose the integer `id`. This keeps URLs stable
across re-imports of the catalog and matches the existing
`/api/problems/:slug` shape.

## Consequences

**Positive**
- Courses are independent of groups and exams. Groups can land in any
  shape later (open-enrolment, instructor-curated, single-cohort) and
  exams can be added without touching courses.
- The owner-vs-admin split lives in one place (`courses.service`),
  not duplicated across routes / future scripts.
- `course_problems.position` is dumb-simple, so the reorder UX is
  whatever the frontend wants — no "swap two rows in a transaction"
  dance.

**Negative**
- "All authenticated → all courses" is too permissive for a real
  multi-cohort university. We accept this *only* until the groups
  module ships and the read endpoints get rewritten with enrolment
  filtering.
- Allowing duplicate `position` values means two problems can tie. If
  the frontend doesn't sort deterministically, the UI may flicker. We
  document `(position, added_at, problem_id)` as the canonical sort.

## Explicit non-goals

- **No** course archive flag, soft-delete, or version history. A
  deleted course is gone — its `course_problems` go with it.
- **No** course-level prerequisites, learning outcomes, or weights for
  gradebook. Those are exam-level / gradebook-level concerns.
- **No** copy-course / fork-course. If a customer wants "use last
  semester's course as a template" we will revisit then.
- **No** group / enrolment / exam / submission-scoped-to-course in
  this commit. They get their own ADRs and commits.

## Future re-evaluation

We revisit if:
- A customer wants two instructors to co-own a course. Likely
  outcome: a `course_owners(course_id, user_id)` table replaces the
  single `owner_id` column.
- The pilot wants visible-only-to-enrolled-students courses *before*
  groups exist. Likely outcome: add `visibility TEXT` ('public' |
  'private') to `courses` as an interim shim until groups land.
