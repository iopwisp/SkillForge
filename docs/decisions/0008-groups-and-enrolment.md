# 0008 — Groups, enrolment, and the narrowed course visibility

- **Status:** accepted (initial implementation shipped — uncommitted)
- **Date:** 2026-05-08

## Context

ADR 0007 deliberately left courses readable by every authenticated user
("must be tightened before AITU pilot ships"). The placeholder works
during prototyping but is wrong for an installation with thousands of
students who shouldn't see each other's curricula.

The university's mental model is:

```
Course "Backend Engineering 2026"
├── Group "MSWE-A"  ← cohort of ~25 students taught Mon/Wed
├── Group "MSWE-B"  ← cohort of ~25 students taught Tue/Thu
└── Group "Retake"  ← students re-taking the course
```

A **group** is one cohort enrolled in a course for a specific term.
Students belong to one or more groups; an instructor's permissions are
attached to the *course*, not to individual groups (one instructor
typically runs all sections of their course).

Once groups exist, the read endpoints can stop showing every course to
every student.

## Decision

### Tables

```sql
CREATE TABLE groups (
  id          INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  course_id   INTEGER NOT NULL REFERENCES courses(id) ON DELETE CASCADE,
  slug        TEXT NOT NULL,
  title       TEXT NOT NULL,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (course_id, slug)
);

CREATE TABLE group_members (
  group_id   INTEGER NOT NULL REFERENCES groups(id) ON DELETE CASCADE,
  user_id    INTEGER NOT NULL REFERENCES users(id)  ON DELETE CASCADE,
  joined_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (group_id, user_id)
);
```

`groups.slug` is unique **per course**, not globally — two different
courses can both have a "section-a". The composite UNIQUE
`(course_id, slug)` enforces this. URLs reflect that:

```
/api/courses/:courseSlug/groups/:groupSlug
```

### Cascade behaviour

- `groups.course_id → courses(id) ON DELETE CASCADE` — deleting a course
  also removes its groups. The semantic of "course is gone" makes the
  groups meaningless.
- `group_members.group_id → groups(id) ON DELETE CASCADE` — same
  reasoning.
- `group_members.user_id → users(id) ON DELETE CASCADE` — when a user
  is deleted, their enrolment rows go with them (this does NOT trigger
  on a normal "remove user from group" call, which is a DELETE on
  `group_members` rows directly). The course owner's user row is still
  protected by the `courses.owner_id NO ACTION` FK from ADR 0007, so a
  user that *owns* a course still cannot be deleted.

### Permissions

| Action                                                              | Required role/relation    |
|---------------------------------------------------------------------|---------------------------|
| `GET …/courses/:c/groups`                                            | requireAuth + see below   |
| `GET …/courses/:c/groups/:g`                                         | requireAuth + see below   |
| `POST …/courses/:c/groups`                                           | course owner OR ADMIN     |
| `PUT …/courses/:c/groups/:g`                                         | course owner OR ADMIN     |
| `DELETE …/courses/:c/groups/:g`                                      | course owner OR ADMIN     |
| `GET …/courses/:c/groups/:g/members`                                 | requireAuth + see below   |
| `POST …/courses/:c/groups/:g/members`                                | course owner OR ADMIN     |
| `DELETE …/courses/:c/groups/:g/members/:username`                    | course owner OR ADMIN     |

Read visibility ("see below"):

| Caller role | Sees                                                        |
|-------------|-------------------------------------------------------------|
| ADMIN       | every group / member                                        |
| INSTRUCTOR  | every group / member (so they can browse other syllabi)     |
| STUDENT     | only groups they are a member of, in courses they're in     |

The "STUDENT sees only their own group" rule is the moral equivalent
of "you can see your classmates but not other sections' rosters". In
practice it also keeps a student from probing the existence of cohorts
they don't belong to.

The instructor-sees-all rule is a Phase 1 simplification. AITU has
~200 instructors; making them see all courses is a non-issue. If a
larger customer asks for "instructors only see their own courses /
their dept's courses", the rewrite is a `WHERE owner_id = $1 OR
co_owner = ...` and lives in courses/queries.

### Course visibility

The visibility change replaces the placeholder from ADR 0007:

| Caller role | `GET /api/courses` returns | `GET /api/courses/:slug` returns |
|-------------|---------------------------|-----------------------------------|
| ADMIN       | every course              | any course                        |
| INSTRUCTOR  | every course              | any course                        |
| STUDENT     | courses they're enrolled in via at least one group | the course iff enrolled, else **404** |

We return **404 not 403** for "not enrolled" so the existence of
unrelated courses doesn't leak through the auth layer.

### Cross-module data flow

The read queries that drive STUDENT visibility (`listCoursesForStudent`,
`isStudentInCourse`) live in `courses/queries.js` and JOIN through the
`groups` and `group_members` tables. This is the same "read-only view
query" pragma that the users module uses to JOIN submissions/problems
on dashboard endpoints — see ADR 0003 §"Allowed dependencies".

The lookups that the groups module needs from the courses table
(`SELECT id, owner_id FROM courses WHERE slug = $1` to validate course
exists + extract owner for the owner-or-admin check) live in
`groups/queries.js` for the symmetric reason. **Neither service
imports the other**, which keeps module boundaries clean and avoids a
circular import at module-load time.

The username → user_id lookup in `groups.queries.findUserIdByUsername`
is also a tiny read-only view query on the users table.

## Consequences

**Positive**
- Students stop seeing other cohorts' curricula. The only Phase-0
  carry-over of "everyone sees everything" is gone.
- The data model maps directly to "section A / section B" without
  introducing tenancy.
- `group_members` is the single place to ask "is user X enrolled in
  course Y?" — used both by the UI ("My Courses") and later by the
  exam module ("can this student take this exam?").

**Negative**
- ADMIN-on-INSTRUCTORs and INSTRUCTORs-on-each-other still see
  everything. We accept that for the AITU pilot scale; it's documented
  as a future tightening if a larger customer asks for it.
- Two students in the same course but different groups can't see each
  other in the members list. If a customer needs a "course roster"
  view (across all groups), we add a separate endpoint scoped to
  course-owner / ADMIN.

## Explicit non-goals

- **No** time-bounded enrolment (start_date / end_date). A student is
  enrolled in a group "now"; archiving a course is "delete the course"
  for now.
- **No** waitlists, group capacity caps, or auto-enrolment rules. A
  human (instructor or admin) adds students one at a time via the API.
  Bulk import lands later.
- **No** per-group instructor / TA. The course owner runs every group.
- **No** joining a group as a side-effect of any user-initiated flow
  (no "enrol myself with code"). Groups are instructor-curated.

## Future re-evaluation

We revisit if:
- A customer wants per-group instructors or TAs. Likely outcome: add
  a `role` column to `group_members` ('STUDENT' | 'TA') or a separate
  `group_instructors` table.
- The cohort sizes pass a few hundred and the "list members" payload
  grows too large. Likely outcome: paginate the members endpoint.
- A customer asks for self-enrolment via codes. Likely outcome: add a
  `enrolment_codes(group_id, code, expires_at)` table; route layer
  handles the redemption flow.
