# 0009 — Exams: window + per-student duration + attempts

- **Status:** accepted (implementation shipped — uncommitted)
- **Date:** 2026-05-08

## Context

Phase 1 needs graded assessments: "on Friday from 10:00 to 12:00 every
student in CS101 section-A solves three problems under a 90-minute
personal timer". That is three real-world constraints on the existing
"student submits code" pipeline:

1. **An availability window** — the exam is live between `starts_at`
   and `ends_at`, wall-clock. Outside that window, no attempts start
   and no submissions are accepted.
2. **A per-student duration** — each student gets a fixed budget
   (90 min) measured from when *they* clicked Start. A student who
   starts 30 min into the window still gets the full 90 min (subject
   to the exam's `ends_at` cap).
3. **One attempt per student per exam** — no retake buttons. A second
   attempt is a separate exam at the instructor's discretion.

This ADR also settles how exam submissions plug into the existing
`submissions` table without duplicating the judge pipeline.

## Decision

### Tables

```sql
CREATE TABLE exams (
  id                INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  course_id         INTEGER NOT NULL REFERENCES courses(id) ON DELETE CASCADE,
  group_id          INTEGER REFERENCES groups(id) ON DELETE CASCADE,
  slug              TEXT NOT NULL,
  title             TEXT NOT NULL,
  description       TEXT,
  starts_at         TIMESTAMPTZ NOT NULL,
  ends_at           TIMESTAMPTZ NOT NULL,
  duration_minutes  INTEGER NOT NULL,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (course_id, slug),
  CHECK (ends_at > starts_at),
  CHECK (duration_minutes > 0)
);

CREATE TABLE exam_problems (
  exam_id    INTEGER NOT NULL REFERENCES exams(id)    ON DELETE CASCADE,
  problem_id INTEGER NOT NULL REFERENCES problems(id) ON DELETE CASCADE,
  position   INTEGER NOT NULL DEFAULT 0,
  points     INTEGER NOT NULL DEFAULT 1,
  PRIMARY KEY (exam_id, problem_id),
  CHECK (points > 0)
);

CREATE TABLE exam_attempts (
  id           INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  exam_id      INTEGER NOT NULL REFERENCES exams(id)  ON DELETE CASCADE,
  user_id      INTEGER NOT NULL REFERENCES users(id)  ON DELETE CASCADE,
  started_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  finished_at  TIMESTAMPTZ,
  UNIQUE (exam_id, user_id)
);

ALTER TABLE submissions
  ADD COLUMN exam_attempt_id INTEGER
    REFERENCES exam_attempts(id) ON DELETE SET NULL;
CREATE INDEX idx_submissions_exam_attempt ON submissions(exam_attempt_id);
```

Key shape choices:

- **`group_id` nullable**. `NULL` means the exam is open to every
  student who is enrolled in the course via *any* group. A non-NULL
  `group_id` restricts the exam to that one group, so a course with
  MSWE-A and MSWE-B can run two separately-timed exams.
- **`UNIQUE (course_id, slug)`**. Exam slug is unique per course; two
  different courses can both have a `midterm`.
- **`submissions.exam_attempt_id` nullable** with `ON DELETE SET
  NULL`. A submission made inside an attempt keeps its row even if
  the attempt is later deleted (rare — cascade is one-way from
  the exam level). Regular practice submissions simply leave the
  column NULL.

### Lifecycle of an attempt

```
Instructor creates exam (POST /api/courses/:c/exams)
  ↓
Exam enters its window  (now >= starts_at, now < ends_at)
  ↓
Student calls POST /api/courses/:c/exams/:e/attempts
  ↓  inserts one `exam_attempts` row with started_at = NOW()
  ↓  the student's personal deadline is
  ↓    min(started_at + duration_minutes, ends_at)
  ↓
Student posts solutions with
  POST /api/courses/:c/exams/:e/attempts/current/submissions/:problemSlug
  ↓  goes through the usual judge/runJudge pipeline
  ↓  the inserted `submissions` row carries `exam_attempt_id`
  ↓  `problem_id` counters + rating bump work as usual
  ↓
Attempt ends either:
  (a) student POSTs .../attempts/current/finish, or
  (b) NOW > personal deadline (computed on read)
  ↓
Score = sum of `exam_problems.points` over problems where this
  attempt has at least one ACCEPTED submission.
```

### Temporal predicates

Three predicates matter:

| Predicate                | Used by                                         |
|--------------------------|-------------------------------------------------|
| `now BETWEEN starts_at AND ends_at` | startAttempt check                   |
| `attempt.finished_at IS NULL AND now < personal_deadline` | submit   |
| `now >= starts_at`       | "mutations on exam / problems are now frozen"   |

These live in `exams.service` as small boolean helpers; they are NOT
pushed into SQL except where an index helps (`idx_submissions_exam_attempt`
is the only one that reads like a join optimisation). Postgres' `NOW()`
is UTC; the service does all comparisons in UTC too.

### Scoring

Score is **computed on demand**, never persisted. A `GET` on the
attempt runs a single query that returns, per problem in the exam,
whether the attempt has any ACCEPTED submission:

```sql
SELECT ep.problem_id, ep.points,
       EXISTS (
         SELECT 1 FROM submissions s
         WHERE s.exam_attempt_id = $1
           AND s.problem_id = ep.problem_id
           AND s.status = 'ACCEPTED'
       ) AS solved
FROM exam_problems ep
WHERE ep.exam_id = $2
```

This keeps the attempt row itself write-once and makes score
recalculation free when the judge becomes more lenient or an
exam's problem list changes. A gradebook-side cache can come later.

### Permissions

| Action                                                            | Required role/relation                     |
|-------------------------------------------------------------------|--------------------------------------------|
| `POST .../exams`                                                  | course owner OR ADMIN                      |
| `PUT .../exams/:slug`                                             | course owner OR ADMIN; exam NOT yet started |
| `DELETE .../exams/:slug`                                          | course owner OR ADMIN                      |
| `POST .../exams/:slug/problems`                                   | course owner OR ADMIN; exam NOT yet started |
| `DELETE .../exams/:slug/problems/:problemSlug`                    | course owner OR ADMIN; exam NOT yet started |
| `GET .../exams`, `GET .../exams/:slug`                            | requireAuth + scope (see below)            |
| `POST .../exams/:slug/attempts`                                   | requireAuth + scope + in-window + no-prior-attempt |
| `POST .../exams/:slug/attempts/current/submissions/:problemSlug`  | requireAuth + own-active-attempt            |
| `POST .../exams/:slug/attempts/current/finish`                    | requireAuth + own-active-attempt            |
| `GET .../exams/:slug/attempts/me`                                 | requireAuth + scope                         |
| `GET .../exams/:slug/attempts/:username`                          | course owner OR ADMIN (or `username == me`) |

**Scope** for a STUDENT reading exams on a course:
- Exams with `group_id IS NULL` are visible to every student enrolled
  in the course (via any group).
- Exams with a specific `group_id` are visible only to members of
  that group.

INSTRUCTORs and ADMINs see every exam in every course, consistent
with Phase 1's other visibility choices (ADR 0008).

**"exam NOT yet started"** is the service-side guard that keeps
instructors from moving the goalposts mid-exam. Once
`starts_at <= now`, the exam's definition (title, problem list,
window, duration) is frozen. DELETE is still allowed because that's
an emergency eject handle.

### Hot-path invariants

- No two simultaneous attempts per student per exam — `UNIQUE (exam_id,
  user_id)` enforces it at the DB level; the service returns 409 if a
  student hits "start" twice.
- Submissions outside the attempt's personal deadline are refused with
  a 400 (not 404) so the UI can display a clear "time is up" message.
- A STUDENT's submissions from a currently-running attempt do NOT leak
  into `GET /api/submissions/recent` so the public feed does not spoil
  answers in-exam. This is a single `WHERE s.exam_attempt_id IS NULL`
  filter in `submissions/queries.getRecentActivity` — we add it in
  this commit.

## Consequences

**Positive**
- One source of truth for submissions (the existing `submissions`
  table). No duplicated judge pipeline, no separate exam-vs-practice
  submission histories to sync.
- The scoring query is a single JOIN — a per-student exam report is
  one round-trip.
- Window + duration cleanly model the real-world constraint without
  introducing a scheduling module.

**Negative**
- One attempt per exam is strict. A customer that wants retakes has
  to run a second exam row. We accept this until a customer explicitly
  asks for per-exam retake policy.
- "Exam frozen once started" blocks `PUT /exams/:slug` as soon as
  `starts_at` passes, even if nobody has started an attempt yet. A
  later commit can relax this to "no attempts exist yet".
- Recent-feed filter on `exam_attempt_id IS NULL` is correct for the
  Phase 1 exam model but will need a rethink when we add proctoring
  or shared classroom mode.

## Explicit non-goals

- **No** proctoring, webcam, screen-sharing, or anti-cheat of any
  kind. Phase 3.
- **No** scheduled auto-start or email reminders. Students manually
  click Start.
- **No** mid-exam time extensions (granting one student extra time
  for accessibility, for example). Future endpoint on attempts.
- **No** randomised problem sets per student. Every student in the
  exam sees the same problems in the same order.
- **No** partial credit / graded subtasks. One problem, one ACCEPTED
  submission, full points. Gradebook module is where weighted grading
  lives.

## Future re-evaluation

We revisit if:
- A customer needs retakes / make-ups. Likely outcome: drop the
  `UNIQUE (exam_id, user_id)` on `exam_attempts` and add a per-exam
  `max_attempts` column.
- Anti-cheat becomes a procurement gate. Likely outcome: a separate
  `proctoring_events` table + webcam integration — orthogonal to the
  exam model itself.
- Scoring gets expensive enough to need caching. Likely outcome:
  materialised view or `exam_attempt_scores(exam_attempt_id,
  total, computed_at)` cache table with recompute hooks.
