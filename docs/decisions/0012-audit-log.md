# 0012 — Audit log

- **Status:** accepted (implementation shipped — uncommitted)
- **Date:** 2026-05-08

## Context

By this point Phase 1 already supports:

- role management
- courses, groups, and enrolment
- exams and gradebook
- instructor problem authoring

That means privileged users can now make changes that materially affect
rosters, assessment setup, and the live catalog. For an on-prem
university installation, we need a basic answer to:

> "Who changed what, and when?"

This ADR settles the first audit-log cut.

## Decision

### Table

```sql
CREATE TABLE audit_events (
  id             INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  actor_id       INTEGER REFERENCES users(id) ON DELETE SET NULL,
  actor_username TEXT NOT NULL,
  actor_role     TEXT NOT NULL,
  action         TEXT NOT NULL,
  entity_type    TEXT NOT NULL,
  entity_key     TEXT NOT NULL,
  details_json   JSONB NOT NULL DEFAULT '{}'::jsonb,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Why snapshot `actor_username` and `actor_role` instead of reading them
only via JOIN later?

- the log stays legible even if the actor is later deleted
- the log preserves the actor's role **at event time**
- read queries stay simple and cheap

### Read API

```text
GET /api/audit-log
```

This endpoint is **ADMIN-only**.

Rationale: the audit log is installation-level operational data, not a
course-scoped teaching surface. Instructors should not automatically see
other instructors' administrative activity.

Basic filters are supported:

- `action`
- `entityType`
- `actorUsername`
- `entityKey`
- `page`
- `pageSize`

### Event surface

Phase 1 logs successful privileged mutations in:

- user role changes
- courses and course-problem attachment changes
- groups and group membership changes
- exams and exam-problem attachment changes
- problem catalog create/update/delete

This is intentionally the **privileged mutation surface**, not every
request in the system.

Examples of event tuples:

| action | entity_type | entity_key |
|---|---|---|
| `SET_ROLE` | `USER_ROLE` | `alice` |
| `CREATE` | `COURSE` | `cs101` |
| `ATTACH` | `COURSE_PROBLEM` | `cs101:http-status-map` |
| `CREATE` | `EXAM` | `cs101:midterm` |
| `DELETE` | `PROBLEM` | `temporary-problem` |

### Write semantics

Audit rows are written **only for successful mutations**.

That means:

- validation failures do not create events
- authorization failures do not create events
- domain conflicts (e.g. "cannot delete referenced problem") do not
  create events

When a mutation already runs inside a transaction, the audit insert is
performed in the same transaction so the event cannot commit without the
state change.

## Consequences

**Positive**

- The installation owner gets a clear, queryable mutation trail.
- Actor snapshots make the log stable even if user data changes later.
- ADMIN-only visibility keeps the operational surface tight.

**Negative**

- The event taxonomy is still hand-curated; adding new mutating features
  means remembering to instrument them.
- The log is append-only and human-readable, but not yet a formal
  compliance product with retention policies, signatures, or export jobs.

## Explicit non-goals

- **No** audit rows for read-only requests.
- **No** student submission spam in the audit log.
- **No** cryptographic tamper-proofing.
- **No** retention/archival policy in this phase.
- **No** per-course instructor-visible audit view.

## Future re-evaluation

We revisit if:

- customers need CSV / SIEM export
- the product gains destructive admin flows like user deletion or data
  restore and needs deeper forensic detail
- instructors later need course-scoped audit visibility separate from
  installation-level admin history
